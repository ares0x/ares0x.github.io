设计原理

O(1) 读写性能，提供键值之间的映射

哈希函数

实现哈希表的关键点在于哈希函数的选择，哈希函数会在很大程度上影响哈希表的读写性能。理想状况下，哈希函数能将不同的键映射到不同的索引上。

比较实际的方式是让哈希函数能够尽量均匀地分布，然后通过工程上的手段解决哈希碰撞问题。

冲突解决

通常情况下，哈希函数输入的范围远大于输出的范围，所以在使用哈希表时一定会遇到冲突。解决哈希碰撞的常用方法：开放寻址法和拉链法。

开放寻址

开放寻址法的核心思想是：依次探测和比较数组中的元素以判断目标的键值是否存在哈希表中（底层数据结构是数组）

1

index = hash("xxx") % array.len

写：向哈希表写入数据时，如果发生冲突，会将键值对写入下一个索引不为空的位置 读：需要查找某个键对应的值时，会从索引位置开始探测数组，直到找到目标元素或者内存为空

开放寻址法中对性能影响最大的是装载因子。 开放寻址法的装载因子是：数组中元素数量和大小的比值 随着装载因子的增加，线性探测的平均用时会逐渐增加，会影响哈希表的读写性能。当装载超过 70% 后，性能急剧下降。一旦装载达到 100%，整个哈希表会完全失效。（查找和插入O(n)）。

拉链法

拉链法是哈希表最常见的实现方法。大多数编程语言采用拉链法实现哈希表。 相较于开放寻址法，实现复杂， 平均的查找长度更短，用于存储节点的内存是动态申请的，比较省存储空间。

拉链法会使用链表数组作为哈希底层的数据结构，可以将其看成可扩展的二维数组。

写： 向哈希表中写入键值对时，键值对中的键回先经过一个哈希函数，哈希函数返回的哈希会帮助我们选择一个桶。选择桶的方式是直接对哈希返回的结果取模。

1

index = hash("xxx") % array.len

当选择了桶之后，就可以遍历当前桶中的链表了，在遍历时会遇到以下两种情况：

    1. 找到键相同的键值对 - 更新键对应的值
    1. 没有找到键相同的键值对 - 在链表的末尾追加新的键值对

读：对该键哈希取模获得对应的桶，遍历该桶中的链表

一个性能较好的链表中，每一个桶中都应该有 0～1 个元素，有时会有 2～3 个，很少超过这个数量。 计算哈希，定位桶和遍历链表三个过程时哈希表读写操作的主要开销。 拉链法的装载因子是：元素数量 / 桶数量

装载因子越大，哈希的读写性能越差。

// src/runtime/map.go 
type hmap struct {
	count     int // 当前哈希表中的元素数量
	flags     uint8
	B         uint8  // 
	noverflow uint16 // approximate number of overflow buckets; see incrnoverflow for details
	hash0     uint32 // hash seed

	buckets    unsafe.Pointer // array of 2^B Buckets. may be nil if count==0.
	oldbuckets unsafe.Pointer // previous bucket array of half the size, non-nil only when growing
	nevacuate  uintptr        // progress counter for evacuation (buckets less than this have been evacuated)

	extra *mapextra // optional fields
}

type mapextra struct {
	overflow    *[]*bmap
	oldoverflow *[]*bmap
	nextOverflow *bmap
}
  • count 表示当前哈希表中元素数量
  • B 表示当前哈希表持有的 buckets 数量，哈希表中桶的数量都是 2 的倍数，所以该字段会存储对数，即 len(buckets) = 2^B
  • hash0 是哈希的种子，作用时为哈希函数的结果引入随机性。该值在创建哈希表时确定，并在调用哈希函数时作为参数传入
  • oldbuckets 时哈希函数扩容时用于保存之前 buckets 的字段，大小是当前 buckets 的一般。

hmap 的桶是 bmap，每一个 bmap 能存储 8 个键值对，当哈希表中存储的数据过多，单个桶已经装满时会使用 extra.nextOverflow 中桶存储溢出的数据

桶（正常桶和溢出桶）在内存中是连续的

bmap 只定义了一个 tophash 字段，该字段存储了键的哈希的高 8 位，通过比较不同键的哈希的高 8 位可以减少访问键值对的次数。

1
2
3

type bmap struct {
    tophash [bucketCnt]uint8
}

bmap不止包含 tophash 字段（跟 slice 类似）。哈希表中可以存储不同类型的键值对，且在 1.18 之前不支持范型，所以键值对占据的内存空间大小只能在编译时进行推导。 bmap 中的其他字段在运行时也是通过计算内存地址的方式访问的，其他字段在 gc.bmap 函数重建结构

type bmap struct {
    topbits [8]uint8
    keys [8]keytype
    values [8]valuetype

    overflow uintptr

哈希表存储的数据逐渐增多，会扩容哈希表或使用额外的桶存储溢出的数据，不会让单个桶中的数据超过 8 个。溢出桶时临时方案，创建过多的溢出桶最终也会导致哈希的扩容

初始化

Go 中初始化哈希的两种方式 - 通过字面量和运行时。

字面量

1
2
3
4

mapItem := map[int]string{
    1:"ares",
    2:"echo",
}

通过字面量初始化 map 需要在初始化时制定键值对的类型，这种方式会通过 gc.maplit 初始化。 如果键值对小于等于 25 个，编译器会将字面量初始化结构转换为

1
2
3

mapItem := make(map[int]string,2)
mapItem[1] = "ares"
mapItem[2] = "echo"

这种初始化方式与数组和切片相同【在 Go 语言汇总，集合类型的初始化有着相同的逻辑】

超过 25 个，编译器会创建两个数组分别存储键和值，这些键值会通过如下所示的 for 循环加入哈希

for i := 0; i < len(vstak); i++ {
    hash[vstatk[i]] = vstatv[i]

运行时

当创建的哈希被分配到栈上切容量小于 BUCKETSIZE = 8 时，Go 在编译阶段会使用

1
2
3
4
5
6
7

var h *hmap
var hv hmap
var bv bmap
h := &hv
b := &bv
h.buckets = b
h.hash0 = fashtrand0()

初始化哈希，这时编译器对最小容量哈希做的优化

只要使用 make 创建哈希，Go 在类型检查阶段会将其转换成 runtime.makemap,使用字面量创建哈希时语言提供的辅助工具，最后调用的都是 runtime.makemap

函数会执行以下步骤：

  1. 计算哈希占用的内存是否溢出或超过能分配的最大值；
  2. 调用 runtime.fastrand 获取一个随机的哈希种子；
  3. 根据传入的 hint 计算出最小需要的桶数量；
  4. 使用 runtime.makeBucketArray 创建用于保存桶的数组。

makeBucketArray 根据传入的 B 计算需要创建的桶数量并在内存中分配一块连续的空间用于存储数据

  • 当桶的数量小于 2^4 时，使用溢出桶的可能性比较低，会省略创建的过程，减少开销；
  • 当桶的数量多于 2^4 时，会额外创建 2^(B-4) 个溢出桶

访问

编译期间的类型检查阶段， hash[key] 会被转换为哈希的 OINDEXMAP 操作，中间代码生成阶段会再进行一次转换

1
2

v  := hash[key] // => v     := *mapaccess1(maptype, hash, &key)
v, ok := hash[key] // => v, ok := mapaccess2(maptype, hash, &key)

mapaccess1 通过哈希表设置的哈希函数，种子获取当前键对应的哈希，再通过 bucketMask 和 add 获取该键值对所在的桶序号和哈希的高 8 位数字。

哈希依次遍历正常桶和溢出桶中的数据，先比较哈希的高 8 位和桶中存储的 tophash，后比较传入的和桶中的值。

当使用 v, ok := hash[key] 时，能够通过布尔值更准确地知道当 v == nil 时， v到地址哈希中存储的元素还是表示该键对应的元素不存在。

写入

使用 hash[key] = xxx 时写入时，表达式会在编译期间转换为 runtime.mapassign 函数

  • 首先根据传入的键拿到对应的哈希和桶
  • 遍历比较桶中存储的 tophash 和键的哈希，如果找到了相同的结果就会返回目标位置的地址。for 循环会依次遍历正常桶和溢出桶中存储的数据。整个过程会分别判断 tophash 是否相等，key是否相同。

如果当前桶已满，会调用 newoverflow 创建新桶或使用 hmap 预先在 noverflwo 中创建好的桶中保存数据。

如果当前键值对在哈希中不存在，哈希回味新键值对规划存储的内存地址。通过 typedmemmove 将键移动到对应的内存空间中并返回键对应的值的地址。如果当前键值对在哈希中存在，那么就会直接返回目标区域的内存地址。

扩容

runtime.mapassign 会在以下两种情况下触发哈希扩容：

    1. 装载因子超过 6.5 （元素数量 / 桶数量）
    1. 哈希使用了太多溢出桶。 哈希扩容不是原子过程，在该方法中还会判断当前哈希是否已经处于扩容状态，避免二次扩容造成的混乱。

不同的触发条件又不同的扩容方式：

  • 如果是溢出桶太多导致，会使用 sameSizeGrow。当我们持续向哈希中插入数据并将他们全部删除时，如果哈希表中的数据量没有超过阈值，就会不断积累溢出桶，造成缓慢的内存泄漏。一旦哈希中出现了过多的溢出桶，它就会创建新桶保存数据，垃圾回收会清理老的溢出桶并释放内存。过程中会通过 makeBucketArray 创建一组新桶和预创建的溢出桶，将原油的桶数组设置到 oldbuckets 上，并将新的空桶设置到 buckets 上。

等量扩容创建的新桶数量和旧桶数量一样。 数据迁移发生在 evacuate 中，它将传入桶中的元素进行再分配。它将一个旧桶中的数据分流到两个新桶，创建两个用于保存分配上下文的 runtime.evacDst 结构体，这两个结构题分别指向一个新桶。

如果是等量扩容，旧桶和新桶之间是一对一关系，evacDst 只初始化一个。如果是翻倍扩容，每个旧桶的元素分流到两个新创建的桶中。

当哈希表的 oldbuckets 存在时，会先定位到旧桶并在该桶没有被分流时从中获取键值对。

扩容总结： 哈希在存储元素过多时会触发扩容操作，每次都会将桶的数量翻倍，扩容过程不是原子的，而是通过 runtime.growWork 增量触发的，在扩容期间访问哈希表时会使用旧桶，向哈希表写入数据时会触发旧桶元素的分流。除了这种正常的扩容之外，为了解决大量写入、删除造成的内存泄漏问题，哈希引入了 sameSizeGrow 这一机制，在出现较多溢出桶时会整理哈希的内存减少空间的占用。

删除

delete 关键字唯一的作用时将某一个键对应的元素从哈希表中删除，无论该键对应的值是否存在，这个内建函数都不会返回任何结果。

如果删除期间遇到了哈希表的扩容，会分流桶中的元素，分流结束之后再找到对应的目标元素完成键值对的删除工作。

哈希表的每个桶都只能存储 8 个键值对，一旦当前哈希的某个桶超出 8 个，新的键值对就会存储到哈希的溢出桶中。
